## `BITS`: target processor mode [nasmdoc.txt:5468-5532]

`BITS 16`/`32`/`64` tells NASM which instruction encodings to emit. The `bin`
output format **defaults to 16-bit mode already** (since it's mainly used for
DOS `.COM`/`.SYS`/boot sectors), so `xfercom.asm`'s explicit `bits 16` is
redundant-but-documenting rather than strictly required â€” it's there so the
file is self-describing without needing to know the `bin`-format default.

Important gotcha the manual calls out explicitly: setting `BITS 32` does
**not** let you use 32-bit instructions in a 16-bit DOS program that still
runs on 16-bit hardware â€” it changes what encodings NASM emits, not what CPU
the resulting binary targets. Getting this backwards produces code that
assembles cleanly but crashes (or behaves nonsensically) on real 16-bit
hardware.

## `CPU`: instruction-set restriction [nasmdoc.txt:5895-5949]

```nasm
cpu 8086    ; used at xfercom.asm:26
```

Restricts assembly to instructions available on the named CPU â€” this is
NASM's enforcement mechanism for the project's "8086/8088-clean" constraint
(`CLAUDE.md` hard constraint #1). Keywords range from `8086` up through
`186`/`286`/`386`/.../`x64`; `CPU DEFAULT`/`CPU ALL` remove the restriction
entirely. If a build fails with an "instruction not supported on selected
CPU" error, the fix is almost always either (a) a shift/rotate with an
immediate count other than 1 (needs 80186+), or (b) `PUSH imm`/`PUSHA`/`POPA`
(also 80186+) â€” see [[8086-instruction-set]] for the full forbidden list.

## Producing `.COM` files [nasmdoc.txt:7731-7806]

```nasm
org 100h
section .text
start:
    ; code here
section .data
    ; initialized data
section .bss
    ; uninitialized data (not written to the file â€” resolved to space past EOF)
```

```bash
nasm myprog.asm -fbin -o myprog.com
```

`.COM` files load at offset `0x100` in their segment (PSP occupies the first
256 bytes) and execute starting there. `.bss` items **do not occupy space in
the output file** â€” the assembler just resolves their addresses to point
past the end of the file, relying on that memory being free (zero-initialized
by DOS's normal loading behavior, but not *guaranteed* zero â€” don't rely on
it). This is exactly why `xfercom.asm`'s BSS layout (`xfercom.asm:1403-1439`)
is declared with `equ` offsets rather than `resb`/`resw` â€” same effect
(no emitted bytes), just computed as compile-time constants instead of
assembler-tracked reservations, keeping the tiny-size constraint auditable
by inspection.

## `ORG` in the `bin` format means something different than in MASM [nasmdoc.txt:6105-6126, 7793-7798]

NASM's `ORG` is *only* a base-address offset applied to internal address
references â€” **not** a MASM-style "jump around in the file and overwrite
already-generated bytes" directive. If porting code from a MASM-family
assembler, do not expect `ORG` to let you backtrack in the output stream.

## Local labels [nasmdoc.txt:1689-1756]

A label starting with a single `.` is scoped to the preceding non-local
label (`label1: .loop:` really defines `label1.loop`). This is exactly the
pattern used throughout `xfercom.asm` (`.h_open`, `.h_data`, `.pa_err`, etc.
â€” all local to `start:`'s scope in effect, though technically each is scoped
to its own preceding top-level label like `parse_args:`). A prefix of `..@`
creates a label immune to the local-label scoping (useful inside macros,
not used in this project). Symbols starting with a double period (`..start`,
`..imagebase`) are NASM-reserved special symbols, distinct from ordinary
local labels.

## `EQU` / `TIMES` [nasmdoc.txt:1052-1103]

- `EQU` binds a label to a constant value, evaluated **once** at the point of
  definition (not re-evaluated per reference) â€” used throughout
  `xfercom.asm`'s BSS layout section.
- `TIMES` repeats an instruction or data declaration N times, where N is
  itself a computable expression (e.g. `times 64-$+buffer db ' '` pads to an
  exact length) â€” not currently used in this project's `.asm`, but idiomatic
  for fixed-size buffer padding if ever needed.

## 16-bit C/Pascal interop [nasmdoc.txt:7828-8387]

Covers calling-convention details (stack-argument order, `BP`-frame layout,
near vs. far function pointers by memory model) for linking NASM object
files with 16-bit C or Pascal compilers. **Not applicable to serial-xfer** â€”
`xfercom.asm` is hand-written, has "no runtime, no linker" by design
(`CLAUDE.md`), and calls its own helpers via a custom register-based
convention documented in the file's header comment (`xfercom.asm:22-24`),
not `cdecl`/`pascal`. Recorded here only so a reader chasing "how do NASM
and C talk to each other" doesn't waste time assuming it applies to this
codebase.
